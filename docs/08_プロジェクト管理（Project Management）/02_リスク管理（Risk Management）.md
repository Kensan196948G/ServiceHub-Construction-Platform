# リスク管理（Risk Management）

## 1. リスク管理方針

プロジェクトのリスクを継続的に識別・評価・対応することで、2026年10月2日の社内リリース目標を達成する。

---

## 2. リスク評価基準

### 発生確率

| 評価値 | 確率 | 説明 |
|--------|------|------|
| 5 | 80%以上 | ほぼ確実に発生 |
| 4 | 60〜80% | 発生しやすい |
| 3 | 40〜60% | どちらともいえない |
| 2 | 20〜40% | 発生しにくい |
| 1 | 20%未満 | めったに発生しない |

### 影響度

| 評価値 | 影響 | 説明 |
|--------|------|------|
| 5 | 致命的 | プロジェクト中止・大幅遅延（1ヶ月以上） |
| 4 | 重大 | 主要マイルストーン未達（2週間以上の遅延） |
| 3 | 中程度 | スケジュール調整が必要（1週間程度の遅延） |
| 2 | 軽微 | 軽微な遅延・品質への影響 |
| 1 | 無視可能 | ほぼ影響なし |

### リスクスコア = 発生確率 × 影響度

| スコア | 対応優先度 |
|--------|----------|
| 15〜25 | 高（即時対応） |
| 8〜14 | 中（計画的対応） |
| 1〜7 | 低（監視のみ） |

## 4. リスク対応計画（上位リスク詳細）

### R-001: AI支援での開発効率が期待を下回る

| 項目 | 内容 |
|------|------|
| リスク原因 | LLMの提案品質不安定、コンテキスト管理の複雑さ |
| 早期警戒サイン | Phase1でのタスク完了率が計画の70%を下回る |
| 予防対策 | AI支援ツールの使い方をPhase1で習熟 |
| 対応計画 | フェーズ間のバッファ（5日）を活用、スコープの優先度B機能を削除 |
| 担当者 | 開発者 |
| 対応期限 | 各フェーズ終了時にレビュー |

### R-004: セキュリティ脆弱性の発見

| 項目 | 内容 |
|------|------|
| リスク原因 | ペネトレーションテストによる脆弱性発見 |
| 早期警戒サイン | CI セキュリティスキャンでHigh以上の脆弱性検出 |
| 予防対策 | 各フェーズでセキュリティレビュー実施、Bandit/Safety定期実行 |
| 対応計画 | Phase5に2週間の修正バッファを確保 |
| 担当者 | 開発者・IT管理者 |
| 対応期限 | Phase5（2026/08/01〜08/31） |

## 5. リスク監視スケジュール

| タイミング | 実施内容 |
|-----------|---------|
| 週次 | リスクログのレビュー・更新 |
| フェーズ終了時 | リスク再評価・新規リスクの識別 |
| 月次 | リスク対応進捗の経営報告 |

---

## 6. コンティンジェンシー計画

### スケジュール遅延時の対応

| 遅延幅 | 対応策 |
|--------|--------|
| 1週間以内 | 週次バッファを使用（残業・週末対応） |
| 2週間以内 | 優先度B機能をスコープ外に移動 |
| 1ヶ月以内 | リリース機能をMVP（最小実行可能製品）に絞り込み |
| 1ヶ月超 | 経営判断でリリース日延期を検討 |
