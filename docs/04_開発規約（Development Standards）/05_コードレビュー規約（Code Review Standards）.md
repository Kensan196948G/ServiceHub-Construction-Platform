# コードレビュー規約

## 概要

コードレビューは品質確保と知識共有を目的とする。本規約では、レビュープロセス・レビュアーの責任・チェックリストを定義する。

---

## レビュープロセス

\`\`\`mermaid
graph LR
    A[機能開発完了] --> B[セルフレビュー]
    B --> C[PRを作成]
    C --> D[CIチェック]
    D -->|失敗| E[修正]
    E --> D
    D -->|成功| F[レビュアー割当]
    F --> G[コードレビュー]
    G -->|修正依頼| H[修正・返答]
    H --> G
    G -->|承認| I[マージ]
    I --> J[developへマージ]
\`\`\`

---

## レビュー担当者・承認要件

| ブランチ | 必要承認者数 | タイムアウト |
|---------|-----------|----------|
| main | 2名 | なし（必須） |
| develop | 1名 | 48時間（期限切れ後は別のレビュアーを割当） |

## コードレビューチェックリスト

### セキュリティ

- [ ] SQLインジェクション対策（ORM使用・パラメータバインド）
- [ ] XSS対策（サニタイズ・エスケープ）
- [ ] CSRF対策
- [ ] 認証・認可チェックが適切か
- [ ] 機密情報（パスワード・トークン）がコードにハードコードされていないか
- [ ] ログに個人情報・機密情報が出力されていないか

### コード品質

- [ ] 命名規則が遵守されているか
- [ ] 関数/メソッドが単一責任原則を守っているか
- [ ] 重複コードがないか（DRY原則）
- [ ] エラーハンドリングが適切か
- [ ] ハードコードされた値がないか（定数・設定化）
- [ ] 不要なコメント・デバッグコードがないか

### テスト

- [ ] 単体テストが追加・更新されているか
- [ ] テストがビジネスロジックを適切にカバーしているか
- [ ] エッジケース・異常系のテストがあるか
- [ ] テストカバレッジが80%以上を維持しているか

### パフォーマンス

- [ ] N+1問題がないか
- [ ] 不要なDBクエリがないか
- [ ] 大きなデータセットへのページネーションが適用されているか
- [ ] キャッシュが適切に使用されているか

### ドキュメント

- [ ] docstring/JSDocが記述されているか
- [ ] API変更時にOpenAPI仕様が更新されているか
- [ ] README/設計ドキュメントが更新されているか

## レビューコメントの書き方

### コメントレベル

| レベル | 意味 | 対応 |
|-------|------|------|
| `[BLOCKER]` | マージ不可な重大な問題 | 必ず修正が必要 |
| `[REQUEST]` | 修正を要求する問題 | 修正が必要 |
| `[SUGGEST]` | 改善提案（任意） | 検討してほしい |
| `[QUESTION]` | 質問・確認 | 返答が必要 |
| `[PRAISE]` | 良い点の評価 | 対応不要 |

### コメント例

\`\`\`
[BLOCKER] SQLクエリに直接ユーザー入力が含まれており、
SQLインジェクションの脆弱性があります。
SQLAlchemyのパラメータバインドを使用してください。

[SUGGEST] この処理はRedisでキャッシュすると
パフォーマンスが改善できそうです。後のタスクで検討してみてください。

[QUESTION] この条件分岐の意図が分かりません。
コメントか変数名で意図を説明していただけますか？
\`\`\`

## レビュー禁止事項

- 個人の好みによる変更要求（コーディング規約に明記されていないもの）
- 不必要に攻撃的・批判的なコメント
- レビューの遅延（担当割当から48時間以内に初回レビューを完了すること）
- LGTM（Looks Good To Me）のみの承認（理由なしの承認禁止）
