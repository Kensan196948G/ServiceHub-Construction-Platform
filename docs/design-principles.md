# Design Principles

`01`〜`03` の要件定義書と詳細設計から、Claude Code 運用でも維持すべき設計原則を抽出したものです。

## 1. 要件から逆算する

- 目的
- 対象ユーザー
- 規格、監査、業務制約
- 受入れ条件

この 4 点を先に固定してから実装する。

## 2. 要件、設計、実装、検証を切り離さない

良い自立開発は、以下を往復できる状態です。

- `requirements.md`: 何を満たすか
- `design_spec.md`: どう作るか
- 実装: どこに入れるか
- テスト: どう確認するか

## 3. 単一の真実を持つ

複数リポジトリ統合や移植では特に以下を明確にする。

- 移植先の主システム
- 移植元の責務
- 廃止対象
- 統合後の API / DB / 監査ログの集約先

## 4. 規格と監査を後付けにしない

元資料では ISO27001、ISO20000、NIST CSF、J-SOX が頻出でした。
そのため設計段階から次を決める。

- 誰がアクセスできるか
- 何が記録されるか
- どこまで保存するか
- どの検証が必須か

## 5. 受入れ基準をテストへ落とす

要件表で終わらせず、最低でも次へ変換する。

- API テスト
- 主要画面や主要フローの確認
- 監査、通知、アラートの確認
- 性能や可用性の目標確認

## 6. README は外向けの真実

元資料の README 群から有効だった点:

- 概要表
- アーキテクチャ図
- 開発フロー図
- セットアップ
- 規格マッピング

過剰更新は不要ですが、外部説明に耐えない README は放置しないこと。
