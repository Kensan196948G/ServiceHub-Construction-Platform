# 統合テスト計画

## 概要

本計画書は、ServiceHub Construction Platform の全モジュール統合テストの計画を定義する。各モジュール間のインターフェース、データフロー、業務シナリオの正常動作を検証する。

---

## テスト範囲

| 対象モジュール | テスト対象インターフェース |
|-------------|----------------------|
| 認証 ↔ 全モジュール | JWT認証・RBAC権限制御 |
| 工事案件 ↔ 日報 | 案件別日報の紐付け・集計 |
| 日報 ↔ 原価管理 | 工数から原価への自動反映 |
| 工事案件 ↔ 写真管理 | 案件別写真の紐付け・検索 |
| 安全品質 ↔ ITSM | ヒヤリハットからインシデント起票 |
| ナレッジ ↔ AI支援 | RAGによるナレッジ検索・回答生成 |

## テストシナリオ詳細（IT-001）

### シナリオ：新規案件作成〜日報提出〜承認

```
前提条件:
- ユーザー（現場所長）がログイン済み
- テスト案件データが未作成

手順:
1. 現場所長が新規案件「テスト工事2026」を作成
   - 期待結果: 案件ID発行、ステータス「計画中」
2. 現場所長が案件のステータスを「進行中」に変更
   - 期待結果: ステータス変更成功、履歴記録
3. 現場監督を案件メンバーに追加
   - 期待結果: メンバー追加成功
4. 現場作業員が日報を作成・提出
   - 期待結果: 日報ステータス「提出済み」、監督に通知
5. 現場監督が日報を承認
   - 期待結果: ステータス「監督承認済み」、所長に通知
6. 現場所長が最終承認
   - 期待結果: ステータス「承認完了」、作業員に通知

合否基準:
- 全手順が正常完了
- ステータス遷移が正確
- 通知が適切なユーザーに送信
- 全データがDBに正確に保存
```

## テスト環境・データ管理

### テストデータ

| データ種別 | 件数 | 作成方法 |
|---------|------|---------|
| テストユーザー | 20名（各ロール） | Factoryボーイ |
| テスト案件 | 10件 | Fixtureデータ |
| テスト日報 | 50件 | Factoryボーイ |
| テスト写真 | 100枚 | サンプル画像 |

### テストデータリセット

```bash
# テスト開始前にデータをリセット
pytest --setup-show tests/integration/
# または
make reset-test-data
```
