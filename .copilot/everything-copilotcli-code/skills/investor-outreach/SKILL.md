# investor-outreach

## 概要

この skill は 投資家や候補先への個別連絡文、フォローアップ文面を整える ための実務向けガイドです。

## 使う場面

- 要件や課題に対して 投資家や候補先への個別連絡文、フォローアップ文面を整える 必要があるとき
- 実装方針、レビュー観点、検証手順を事前に揃えたいとき
- 関連する agent や command を組み合わせて一貫した進め方を作りたいとき

## 想定入力

- 対象となるコード、設計書、README、Issue、差分
- 現在の制約条件、使用技術、既存パターン
- 必要ならテスト結果、ビルドログ、利用者要件

## 期待する出力

- 何をどう進めるかの判断基準
- 変更時の注意点、レビュー観点、検証観点
- 実装後に残すべきドキュメントや再開メモ

## 標準手順

1. まず既存コード、仕様、README、関連 Issue を確認する
2. 今回のゴールと非ゴールを 1 回言語化する
3. 既存パターンを壊さない最小変更案を決める
4. 実装またはレビューを進める
5. 変更近傍テスト、lint、build、必要な統合確認を行う
6. 結果を README や変更サマリへ反映する

## 詳細ガイド

### 設計時に確認すること

- 境界が曖昧になっていないか
- 既存の命名、責務分割、設定方法と一致しているか
- 将来の保守や横展開に無理がないか

### 実装時に確認すること

- 最小変更で目的を達成しているか
- 不要な抽象化や過剰最適化を入れていないか
- エラー処理、ログ、例外、入力検証が不足していないか

### 検証時に確認すること

- 変更近傍のテストがあるか
- 主要フローに回帰がないか
- ドキュメントと実装にズレがないか

## 相性のよい command

`/update-docs`

## 相性のよい agents

`chief-of-staff`

## 注意点

- この skill は実装を急がせるためではなく、失敗率を下げるために使う
- 外部状態が読めない場合は推測せず、その前提を明示する
- 大きな変更は 1 回で終わらせず、計画、実装、検証に分けて扱う
